# Sprint 10 — 임/이 간호사 모자 추가 + 다음 버튼 위로 살짝

> 2026-05-21 작업 학습 노트. 중학생도 이해할 수 있게 쉬운 비유로 풀어 씁니다.

## 0. 한 줄로 뭘 했나?

- 캐릭터 선택창에서 **임간호·이간호 머리에 간호사 모자**가 안 보였던 걸 채워줬어요.
- "다음" 버튼이 화면 바닥에 너무 붙어 있어서 **24픽셀쯤 위로 올려** 숨통이 트이게 했어요.

## 1. 왜 이런 일이 벌어졌나?

5명의 캐릭터(김·정·건·임·이) 얼굴을 그리는 함수가 캐릭터마다 따로 있어요.
공통으로 쓰는 부품 함수 `buildNurseCap()` (간호사 모자 그려주는 함수)이 있는데, **임·이 함수는 이걸 호출하는 줄이 빠져 있던** 것이 원인이에요.

비유하자면 — 5명 분 도시락을 싸는데, 김치·정수·건우 도시락에는 "계란말이 한 칸 추가" 라벨이 붙어 있고, 임꺽정·이순신 도시락만 라벨이 빠져 있어서 계란말이 안 들어간 거예요. 라벨 한 줄만 붙이면 끝.

## 2. Spring 비유 — 공통 컴포넌트 호출

Spring으로 따지면 `buildNurseCap()`은 `@Component` 로 등록된 재사용 가능한 빈(bean)이에요.

```java
// Spring
@Component
public class NurseCapRenderer { ... }

// 다섯 컨트롤러가 같은 빈을 주입받지만,
// LeeController·ImController만 @Autowired NurseCapRenderer cap; 를 안 쓰고 있던 셈
```

해결도 똑같아요 — **"의존성을 안 쓰던 곳에서 쓰기 시작"** 한 줄. 빈 자체(공통 함수 본문)는 건들지 않아요. 만약 본문을 고치면 김·정·건도 같이 영향받으니까요.

## 3. SpriteKit의 "동시 zPosition" 트릭 — 호출 순서가 곧 그리기 순서

SpriteKit에서 노드가 어느 게 위에 그려질지는 `zPosition` 숫자로 정해져요. 큰 숫자가 위로 와요.

그런데 **두 노드가 같은 zPosition을 가지면**? 그땐 누가 위인지가 모호해 보이지만, 사실 **나중에 addChild된 게 위로** 그려져요. (HTML/CSS의 `z-index` 동률 처리와 같은 규칙)

이 게임에서:
- 임간호의 **고양이귀**는 zPosition = 20
- 추가하려는 **간호사 모자**도 zPosition = 20

만약 모자를 귀보다 먼저 addChild하면 귀가 모자를 덮는 사고가 나요. 그래서 코드를 이렇게 배치했어요:

```swift
// (귀를 먼저 그림 — zPos=20)
addChild(earNode)
addChild(innerNode)
...
// 그 다음에 모자 호출 — 같은 zPos=20이지만 나중이라 위에 그려짐
buildNurseCap()
```

비유하자면 — 같은 책상 위에 책 두 권을 쌓는데, **나중에 올린 책이 위에 있는 것**과 같아요. 책상 칸수(zPosition)가 같으면 올린 순서가 결국 답이에요.

## 4. "다음 버튼이 너무 아래" — 어떻게 위로 올렸나?

버튼 위치를 정하는 식은 이렇게 생겼어요:

```
buttonY = (화면 바닥)
        + (노치 보정)
        + (장치별 자동 여백)
        + (이 변수가 우리 손잡이)  ← characterSelectConfirmButtonBottomInset
```

그래서 **이 손잡이 값을 40에서 64로 키우면** (= 바닥에서 더 멀어지면) 버튼이 위로 올라와요.

```swift
// 변경 전: 40 — 버튼이 바닥에 붙음
// 변경 후: 64 — 24픽셀 위로 호흡
static let characterSelectConfirmButtonBottomInset: CGFloat = 64
```

## 5. 안전장치 — "버튼이 위 칩이랑 너무 가까워지면 자동 멈춤"

근데 무한정 올릴 순 없어요. 버튼 위에는 "스킬 설명 칩"이 있어서 너무 위로 올라가면 둘이 겹쳐요. 그래서 코드에는 이런 안전판이 있어요:

```swift
let buttonY = min(baseY, maxAllowedY)
// baseY: 우리가 올리고 싶은 위치
// maxAllowedY: 칩이랑 부딪히기 직전 위치
// min(): 둘 중 더 아래쪽을 골라서 — "올라가다가 칩 만나면 멈춰"
```

비유하자면 — 엘리베이터에 "12층까지만 가도록" 자동 멈춤 장치가 달린 거예요. 사용자가 99층 버튼을 눌러도 12층에서 정확히 멈춰요.

산수로 계산해보니 이번엔 24픽셀 정도 올려도 안전판 안 발동해요. 그래서 한 줄만 고치면 끝.

## 7. 같이 따라오는 보너스 — 결과창 미니 얼굴도 모자 생김

`CharacterFaceNode`는 캐릭터 선택창뿐 아니라 **결과창(ScoreboardScene)에서 점수 옆에 작게 뜨는 얼굴** 에서도 똑같이 쓰여요. 같은 함수를 통해 만들어지니까, 이번 변경으로 **결과창의 임·이 얼굴에도 자동으로 모자가 생겨요**.

이건 "의도된 부수효과"예요 — 사용자가 "5명 시각 일관성" 을 원했으니 결과창에서도 일관되는 게 자연스러워요. 만약 결과창은 모자 빼고 싶었다면 그땐 mini용 분기를 따로 두는 작업이 필요했을 거예요.
