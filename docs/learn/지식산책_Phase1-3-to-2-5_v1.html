내가 박스를 게임 캐릭터답게 만들고 싶어 물리엔진을 처음 켠다고 해보자. 1-1에서 만들어둔 PhysicsCategory가 드디어 깨어난다.

position += 명령형 → velocity = 선언형
// 위치 통제를 엔진에게 위임 physicsBody?.velocity = CGVector( dx: dir.dx * speed, dy: dir.dy * speed)
핵심: 인프라 먼저 정의(1-1), 활성화는 나중(2-2) — Spring 패턴

충돌·수집
collision ≠ contact
내가 박스가 음표를 통과하면서도 *닿았다는 알림*은 받고 싶다고 해보자. 같은 SKPhysicsBody에 비트마스크 두 개로 분리된다.
collision: 물리적 막힘 — DB CHECK 제약
contactTest: 충돌 알림 — Event 발행
음표는 collision=0 + contactTest=.player
// 통과 + 알림 body.collisionBitMask = 0 body.contactTestBitMask = .player
핵심: 같은 객체에 *제약*과 *이벤트*를 분리 — DDD SoC 원칙

8단계가 알려준 4가지
Phase 2-4에서 HUD(점수+45초)로 게임이 *세션*이 됐고, 2-5에서 콤보 ×2로 *연속 압박이 보상*인 구조로 진화했습니다.
1. 명령형 → 선언형 전환 (position → velocity)
2. 도메인 vs 씬 책임 분리
3. 매직 넘버 0 + 임시물에 폐기 시점 명시
4. 1-1 자산의 지연 활성화 패턴

CAPTION
SpriteKit으로 만드는 작은 회피 게임, Phase 1-3부터 2-5까지 8단계를 정리해봤습니다. 자동으로 왕복하던 박스가 D-Pad로 움직이고, 맵 경계가 정의되고, 카메라가 드론처럼 따라오는 단계까지가 1-3~1-5였습니다. 이후 외곽 벽이 보이고 SKPhysicsBody가 켜지면서 박스에 *몸*이 생겼고, 음표를 통과하면서도 *닿았다는 알림*은 받는 collision/contact 분리 패턴을 만났습니다. 마지막엔 HUD와 45초 타이머, 콤보 ×2 보상까지 — 게임이 *공간*에서 *세션*으로, 다시 *연속 압박이 보상*인 구조로 진화했습니다. 가장 인상 깊었던 건 같은 SKPhysicsBody에 비트마스크 두 개로 *제약*과 *이벤트*를 분리하는 설계였습니다. Spring DDD에서도 같은 객체에 불변식과 도메인 이벤트를 함께 두는 패턴이 있는데, 게임 엔진에서도 똑같은 SoC 원칙이 보였습니다. 아직 모르는 게 많지만, 오늘도 한 개념씩 쌓아가겠습니다 💪 저장해두고 천천히 보세요.
