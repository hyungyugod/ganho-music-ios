# 15 · Phase 2-9 · F 발사 주기도 점점 빨라진다 ⏱️

> **이번 작업 한 줄**: 게임이 진행될수록 수간호사가 *F를 더 자주* 던진다. 시작은 3.5초마다, 끝(45초 후)은 2초마다.

---

## 1. 게임이 어떻게 변하나?

**전 단계(2-8)까지**: 수간호사 *속도*는 점점 빨라지는데, *F 발사 주기*는 항상 3.5초 그대로.

**이번 단계(2-9)**: F 발사 주기도 *점점 짧아짐*. 끝나갈 무렵엔 *훨씬 자주* F가 날아옴.

```
시간:    0초 ─────────────────── 45초
주기:   3.5 ━━━━━━━━━━━━━━━━━ 2.0
        (느긋)              (빠르게 연속)
```

수간호사 속도(2-8) + F 발사 주기(2-9) 둘 다 *시간 따라* 빨라지면, 끝나갈수록 게임이 *훨씬 어려워짐*. 진짜 *난이도 곡선* 완성.

---

## 2. 어떻게 *주기*를 점점 짧게?

### 2-1. 이전 방식의 한계 (2-7)
```swift
SKAction.repeatForever(.sequence([wait(3.5), run(fireProjectile)]))
```
- "3.5초 기다리고 발사를 영원히 반복"
- 문제: *3.5초가 고정*. 변경 불가.

### 2-2. 새 방식 — *매번 다음 시간을 직접 계산*
```
[발사] → [현재 시간으로 다음 주기 계산] → [그 시간만큼 기다림] → [발사] → 반복
```

매 발사 *직후*에 "지금이 게임 어디쯤이지?" 보고 *다음 주기*를 계산.

### 2-3. 비유: 알람 시계
- 고정 알람: 매일 7시 (이전 방식)
- 적응형 알람: 발사 후 *오늘 일정 보고* "다음은 5시간 후" 또는 "3시간 후"로 재설정 (새 방식)

게임에서:
- 시작 (남은 시간 45초): 다음 발사 *3.5초 후*
- 중간 (남은 시간 22.5초): 다음 발사 *2.75초 후*
- 끝 직전 (남은 시간 0초): 다음 발사 *2.0초 후*

---

## 3. 보간 공식 한 번 더

이전(2-8)에 배운 거 그대로. *두 값 사이를 비율로 채움*.

```
진행률 t = 1 - 남은시간 / 45
주기 = 3.5 + (2.0 - 3.5) × t
     = 3.5 + (-1.5) × t
```

t = 0이면 주기 3.5 (시작값)
t = 0.5면 주기 2.75 (중간)
t = 1이면 주기 2.0 (끝값)

**같은 보간 패턴**, 다만 *수간호사 속도*는 *증가*하고 *F 주기*는 *감소*. (끝값이 시작값보다 작으면 -방향)

## 5. 새로 배운 것 (쉬운 설명)

### 5-1. **재귀 액션이란?**
> 액션이 끝나면 *자기 자신을 다시* 등록.

```swift
private func scheduleNextFire() {
    let wait = SKAction.wait(forDuration: 현재주기())
    let fire = SKAction.run { [weak self] in
        self?.fireProjectile()
        self?.scheduleNextFire()    // ← 자기 자신 다시 호출!
    }
    self.run(.sequence([wait, fire]), withKey: "fireProjectiles")
}
```

흐름:
1. 처음에 `scheduleNextFire()` 호출 → 액션 등록
2. 3.5초 기다림 → fire 클로저 실행
3. F 발사 → 다시 `scheduleNextFire()` 호출 → *새 액션* 등록 (이번엔 새 주기로 wait)
4. 무한 반복

매번 *현재 상태*를 보고 *다음 주기*를 계산하니 *동적*. 이전 `repeatForever`는 *고정 주기*만 가능.

### 5-2. **`[weak self]`의 역할**
> 클로저가 *영원히 self를 잡고 있으면* 메모리 누수.

```swift
SKAction.run { [weak self] in
    self?.fireProjectile()
    self?.scheduleNextFire()
}
```

- `[weak self]` = "self를 *약하게* 참조해. 만약 self가 사라지면 nil로 둬"
- `self?.` = "self가 살아있으면 호출, 죽었으면 그냥 무시"

게임이 *재시작*되거나 *씬이 교체*될 때 GameScene이 사라져야 하는데, 클로저가 *강하게* 잡고 있으면 못 사라짐. weak로 풀어줘야 함.

> Spring으로 치면 ApplicationContext가 destroy될 때 bean들이 정상 해제되는 거랑 같음.

### 5-3. **재귀 vs `repeatForever` 언제 뭐 쓰나?**
| 상황 | 어느 쪽? |
|---|---|
| *고정 주기* 반복 | `repeatForever` (단순) |
| *매번 다른 주기* 필요 | 재귀 (유연) |
| 무한 반복인데 *조건부 중단* | 재귀 (가드 추가 가능) |

### 5-4. **상수 이름 — 시작값 vs 끝값**
```swift
projectileFireInterval: TimeInterval = 3.5      // 시작값 (이전부터 사용 중)
projectileFireIntervalEnd: TimeInterval = 2.0   // 끝값 (이번 신설)
```

기존 `projectileFireInterval`을 *그대로* 두고 *새 상수만* 추가. 이름 변경하면 *기존 호출처*도 수정 필요 → 회귀 위험. *조용히 의미만* 시작값으로 강화 (주석으로).

## 9. 회고

### 9-1. 막혔던 것
**전혀 없음.** 1차 빌드 성공 + **만점 10.0/10** 합격. 변경 표면 +30/-7 줄로 작음. SPEC §"기능 상세"에 코드 스켈레톤 명시 → Generator 정확히 옮김.

### 9-2. 새로 배운 것
1. **재귀 SKAction 패턴** — 액션이 끝나면 *자기 자신을 다시 등록*. SKAction.repeatForever는 *고정 주기*만 가능, 재귀는 *매번 다른 주기* 가능.
2. **`[weak self]` 캡처의 의미** — 클로저가 *영원히 self를 잡으면 메모리 누수*. 약한 참조로 풀어줘서 GameScene이 정상 해제 가능.
3. **withKey 동일 재등록** — SpriteKit이 같은 키로 액션 등록 시 *기존 액션 교체*. 시퀀스 끝난 후 재등록이라 충돌 없음. endGame의 removeAction이 한 곳에서 정지 가능.
4. **콜 스택 누적 없음** — 재귀처럼 보이지만 *액션 클로저 안*이라 비동기. 일반 재귀 함수와 달리 *스택 폭발* 없음.
5. **보간 패턴 일관성** — 2-8 EnemyNode speed 보간과 *동형* 공식. 한 번 배운 패턴을 *다른 값*에 그대로 적용.

### 9-3. 다음으로 미룬 것
1. **Systems 폴더 분리**: GameScene이 더 비대해짐. SpawnSystem / ContactRouter로 분리 필요.
2. **무적 시간 (Phase 5)**.
3. **F 시각 효과 / 사운드 (Phase 6)**.
4. **난이도 normal / hard (Phase 4)**: F 동시 수 3개·4개 등.
