# 학습 노트 작성 스타일 가이드

> `docs/learn/NN-*.md` 학습 노트를 쓸 때 따르는 규칙.
> 한 문장 요약: **"중학생 친구에게 설명한다고 생각하고 시작 → 그 다음 기술 구현 깊이를 붙인다."**

---

## 0. 가장 중요한 원칙

학습 노트의 **앞부분 1~2 섹션**을 읽고도 *코드를 한 줄도 모르는 사람*이:
- 이번 작업이 게임 화면에서 어떻게 보이는 변화를 만드는지
- 왜 이게 필요한지
- 결정 사항이 뭔지

를 이해할 수 있어야 한다. 코드/Swift/SpriteKit 용어는 **§3 이후**에서 등장.

> 사용자는 Spring 백엔드 베테랑이지만 Swift/iOS 첫 학습자. 그래서 *큰 그림*은 캐주얼하게, *기술 구현*은 깊이 있게 — 두 톤을 섞는다.

## 2. 톤별 작성 요령

### 2-1. 중학생 톤 (§1, §2, §6 시각, §7)
**OK 예시**:
> "지금 박스가 D-Pad로 움직이는데 맵 밖 검은 영역까지 슝슝 나가버려요. 운동장에서 펜스 통과해 도로까지 나가는 학생 같이."

**NG 예시**:
> "PlayerNode의 position이 GameConfig.mapWidth/Height의 boundary를 초과하면 viewport에서 clipping이 발생합니다."

**규칙**:
- 비유 1개 이상 (운동장/펜스/카메라맨/슈퍼마리오/택배 분류 같은 일상)
- ASCII 그림으로 변경 전/후 시각화 — 코드 없이 *보이는 결과*만
- "…해요", "…네요" 같은 친근한 어미 OK
- 영어 약어(VC, SDK, IoC) 최소화. 써야 하면 풀어 쓰기

### 2-2. 기술자 톤 (§3, §4, §5, §8, §9)
**OK 예시**:
> "`SKSpriteNode` 상속 + `super.init(texture: nil, color:size:)` designated initializer + `required init?(coder:)`의 `fatalError`는 NSCoding 호환 의도 표명이라 강제 언래핑 카운트와 별개."

**규칙**:
- Swift/SpriteKit 정확한 용어
- 코드 스니펫(```swift) 적극 사용
- Spring 비유는 *Java/Spring 패턴 이름*까지 정확히 (예: "anemic vs rich domain model", "DTO/Entity 분리", "@RestController")
- 함정/오해 가능성을 미리 짚기 (예: "헷갈리는 부분 — `!`가 옵셔널 강제 언래핑이 아니라 Bool 부정인 경우")

## 3. ASCII 그림 사용

학습 노트는 코드보다 **그림으로 시작**한다.

### 변경 전/후 비교
```
지금                          이번 작업 후
┌──────────────┐             ┌──────────────┐
│ (현재 상태)   │     ──→     │ (목표 상태)   │
└──────────────┘             └──────────────┘
```

### 노드 트리
```
GameScene
├── worldNode
│   └── PlayerNode  ← 이번에 신설
└── cameraNode
    └── DPadNode    ← 이번에 신설
```

### 의존성 방향
```
GameScene  →  PlayerNode  →  GameConfig
GameScene  →  DPadNode    →  (외부 의존 0)
```

ASCII 그림 한 장이 코드 10줄보다 빠르다.

## 4. 결정 사항 작성법 (§7)

**모든 옵션을 비유로 풀고 추천을 명시.**

### 표 템플릿
| # | 결정 | 옵션 | 추천 |
|---|---|---|---|
| ① | (질문 한 줄) | A. 비유로 설명 / B. 비유로 설명 | **A** — 한 줄 이유 |

### 본문 템플릿
```markdown
### 결정 ① · (질문)

| 옵션 | 비유 | 결과 |
|---|---|---|
| **A. xxx** ⭐추천 | (일상 비유) | (장점) |
| B. yyy | (일상 비유) | (단점) |

**왜 A?** (한 줄 이유 — 학습 노트의 다른 결정과 일관성 / Spring 패턴과 부합 / 스코프 단순화 등)
```

옵션 비교를 **수치**로 보여줄 수 있으면 더 좋음 (예: "iPhone 17 가로 viewport 852 vs 맵 가로 640 → 화면이 더 큼").

## 5. 기술 구현 (§3-§4) 작성법

### Spring 비유 (§3)
*같은 개념의 양면*:
```
Swift/SpriteKit 쪽           Spring 쪽
SKSpriteNode 상속      ↔    @Entity Order extends BaseEntity
update(deltaTime:) 메서드  ↔    Order.recalculateTotal() (rich domain)
private(set) var       ↔    public field with private setter
cameraNode 자식 좌표   ↔    DTO (응답 시 도메인을 변환)
```

표 형식이 가장 빠르게 읽힘.

### Swift 학습 포인트 (§4)
4-1, 4-2, 4-3 형태로 **개별 학습 포인트 1개씩**:

```markdown
### 4-1. (학습 포인트 제목)

코드:
```swift
(짧은 코드)
```

**왜 이렇게?** (Spring/Java로는 어떻게 했는지 + Swift는 왜 다른지)

**함정**: (실수하기 쉬운 점)
```

학습 포인트 5~8개가 적정. 너무 많으면 본말전도.

## 6. SPEC에 들어갈 핵심 제약 (§8) 작성법

§7 사용자 결정 사항을 흡수하여 **Planner가 SPEC.md로 옮길 수 있는 형태**로 정리:

```markdown
- **변경 유형**: (게임플레이 / 비주얼 / 혼합)
- **게임 경험 의도**: (2~3문장)
- **Sprint 범위 계약**:
  - **IN**: (이번 작업에 포함되는 변경)
  - **OUT**: (이번 작업에 포함 안 됨 — 위반 시 자동 감점)
- **준수 룰**: (강제 언래핑 0, Timer 0, ...)
- **회귀 보존**: (직전 Phase 합격 자산)
```

§8을 잘 쓰면 Planner가 SPEC.md를 *거의 그대로 옮긴다*. 학습 노트가 SPEC의 입력.

## 7. 회고 (§9) 작성법

**작업 직후, QA_REPORT.md 점수 받자마자** 채운다. 빈 칸으로 두지 말 것.

| 부섹션 | 채울 내용 |
|---|---|
| 9-1. 막혔던 것 | Generator가 어디서 헤맸는지, 빌드 에러 / pbxproj 함정 / 좌표계 헷갈림 등 |
| 9-2. Spring과 다르네 싶었던 것 | 사용자 학습 자료 — 이번에 처음 만난 Swift 개념 5~8개 |
| 9-3. 다음 작업 이월 결정 | 미뤄둔 / 추후 다시 봐야 할 항목 |
| 9-4. 평가 점수 | QA_REPORT.md 카테고리별 점수 + 가중평균 |
| 9-5. 사용자 직접 확인할 것 | 시뮬레이터 시각 검증 체크리스트 |

**핫픽스/버그 발견 시**: §9-1에 *수정 방법까지 포함된 회고*를 추가 (Phase 1-3의 §9-1-bis 참고). 다음 사람(미래의 나/Claude)이 같은 함정 안 밟게.

## 8. 파일명 규칙

```
NN-phase-X-Y-슬러그.md
```

- `NN`: 두 자리 시퀀스 (`01`, `02`, ...). 작성 순서.
- `phase-X-Y`: 로드맵의 phase 번호.
- `슬러그`: 짧은 영어 식별자 (`config-bootstrap`, `world-camera`, `player-dpad`, `bounds-clamp`).

예시:
- `01-phase1-1-config-bootstrap.md` ✅
- `02-phase1-2-world-camera.md` ✅
- `03-phase1-3-player-dpad.md` ✅
- `04-phase1-4-bounds-clamp.md` ✅

`STYLE.md`처럼 **메타 문서**는 NN 접두사 없이 대문자로.

---

## 9. 작성 워크플로

```
[1] 사용자가 다음 작업 요청
[2] 학습 노트 NN-*.md 작성 (이 STYLE.md 따라)
    ├─ §1-§2: 중학생 톤 (5분이면 작성)
    ├─ §3-§4: 기술 구현 (사용자가 새로 배울 개념 정리)
    ├─ §5-§8: SPEC 입력 준비
    └─ §9-§10: 회고는 빈칸, 다음 단계 안내
[3] 사용자에게 §7 결정 사항 OK 받기
[4] 하네스 (rm → Planner → Generator → Evaluator)
[5] 합격 시 §9 회고 채우기 + 로드맵 갱신
[6] 불합격 시 Generator 재호출 (최대 3회)
```
