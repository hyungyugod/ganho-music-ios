# Sprint 9 Phase D — 결과창 V4 spacing

## 한 줄로

결과창은 **5층 케이크처럼 위/아래 두 묶음으로 나누는 게 시각 호흡**이다. 위 묶음(이름표·제목·부제·점수)은 모두 위로 끌어올리고, 아래 묶음(가로줄·통계)은 그 자리 그대로 두어, 가운데에 **74pt 빈자리**를 만들었다.

## 무슨 문제를 풀었나

이전 결과창(V3)은 화면 한가운데에 11줄을 욱여넣었다. 위쪽 5줄(난이도 알약 / 강조선 / "실습 종료" / 부제 / "♪ 0")이 너무 가까워서, 글자끼리 거의 닿아 있는 답답한 모양이었어. 빨래를 빨래줄에 빨래집게 없이 5개를 한 손바닥 위에 쌓아둔 느낌이라고 보면 돼.

## 핵심 비유 — 5층 케이크

결과창을 **5층 케이크**라고 생각해보자. 케이크 한 판을 위/아래 두 덩어리로 나누면:

- **위 덩어리** (정체성 정보): 난이도 알약 / 강조선 / 제목 / 부제 / 점수
- **아래 덩어리** (통계): 가로 구분선 / PLAYS 횟수 / TOTAL 합산

문제는 V3에서는 이 두 덩어리가 너무 가깝게 붙어 있었다는 거야. 그래서 V4에서는:

1. **위 덩어리 전체를 +20~30pt 위로 끌어올린다**
2. **아래 덩어리는 거의 그대로 둔다**

그러면 가운데에 자연스럽게 **74pt의 빈자리**가 생긴다. 빈자리가 있어야 사람 눈이 "아, 이 위는 한 묶음이고 저 아래는 다른 묶음이구나"라고 알아챈다.

## 숫자로 보면

| 줄 | V3 (전) | V4 (후) | 차이 |
|---|---|---|---|
| 강조선 | +148 | +178 | **+30 위로** |
| 난이도 알약 | +115 | +145 | **+30 위로** |
| (레거시 제목) | +85 | +100 | +15 위로 |
| 부제 | +58 | +64 | +6 위로 |
| **점수** | -2 | **+6** | **+8 위로** |
| **— 빈자리 74pt —** | | | |
| **가로 구분선** | -78 | **-68** | +10 위로 |
| PLAYS 숫자 | -98 | -96 | (구분선 따라) |

점수(+6) ↔ 구분선(-68) 사이가 **74pt** → 사람 눈이 보기에 명확하게 두 묶음으로 갈라진다.

## Spring 출신을 위한 비유

Spring으로 비유하면 이렇다:

- **GameConfig.swift** = `application.yml` 같은 설정 파일. 모든 좌표값 상수를 모아둔 곳.
- **layoutLabels()** = `@PostConstruct` 메서드 같은 초기화 메서드. 화면이 그려질 때 호출돼서 모든 라벨 위치를 잡아준다.
- **V3 / V4 상수** = `application-dev.yml` / `application-prod.yml`처럼 **다른 버전의 같은 설정**. V3 값을 지우지 않고 V4를 *추가*하는 이유는, 혹시 다른 어디서 V3를 참조하고 있을 때 죽지 않게 하려는 것 + 회귀 안전망.

```swift
// V3 그대로 보존 (다른 곳 참조 가능성)
static let resultHeaderChipOffsetYV3: CGFloat = 115

// V4 새로 추가 (이번에 layoutLabels에서 실제로 사용)
static let resultHeaderChipOffsetYV4: CGFloat = 145
```

## 가장 중요한 약속 — "건드리지 마라" 영역

이번 작업은 **좌표 9줄만** 바꾸는 것이지, **다른 어떤 것도 바꾸지 않는다**:

- `init(finalScore:, personalBest:, difficulty:, characterName:, ...)` 9개 인자 시그니처 → 그대로
- `scoreLabel.text = "\(finalScore)"` → 그대로
- `bestLabel.alpha = 0` (안 보이게 하는 정책) → 그대로
- `DiplomaOverlayNode.present(...)` 호출 시점 → 그대로
- `haptics.heavy()` / `comboMilestoneStrong` 사운드 발화 시점 → 그대로
- `touchesBegan(_:with:)` 터치 처리 전체 → 그대로
- 3개 버튼(기록 보기 / 공유 / 다시 시작) 위치 → 그대로

이 약속을 지키지 않으면, 결과창에 새로 최고점 났을 때 졸업장이 안 뜬다거나, 공유 버튼이 안 눌린다거나 하는 버그가 생긴다. 그래서 **변경 영역을 좌표 9줄 + 새 상수 7개로 명확히 격리**한 거야.

## 상대식의 의미

코드에서 통계 좌표는 다음과 같이 적었다:

```swift
let statValueY = GameConfig.resultDividerOffsetYV4 - GameConfig.resultStatGapFromDividerV9
let statTitleY = statValueY - 14
```

이건 "통계 숫자는 **가로 구분선에서 28pt 아래**, 통계 라벨은 **그 숫자에서 14pt 아래**" 라고 *의도*를 코드 그대로 표현한 거야. 만약 나중에 가로 구분선 위치를 더 위로 옮기면, 통계도 자동으로 따라 올라간다. 절대값 `-98`이라고 하드코딩하면 매번 같이 손 봐줘야 하니까, 상대식이 깔끔하다.

Spring으로 보면 `@Value("${app.divider.y} - ${app.statGap}")` 비슷한 느낌. 의존 관계가 명확해진다.

## 검증 — 어떻게 "잘 됐는지" 알 수 있나

3가지로 확인한다:

1. **빌드 성공**: `xcodebuild ... build` → `** BUILD SUCCEEDED **`
2. **git diff 안전성**: `init` / `touchesBegan` / `DiplomaOverlayNode.present` 등 보호 키워드 grep → 0줄 변경
3. **호흡 산술**: 점수(+6) ↔ 가로 구분선(-68) = **74pt ≥ 60pt** → SPEC 합격 기준 충족

## 이번에 안 한 것

- 시뮬레이터에서 실제로 띄워보고 눈으로 확인 — 사용자 본인이 직접 빌드해서 검증 (시뮬레이터 실측이 진짜 진실).
- 작은 폰트 환경(Dynamic Type 큰 글씨) 대응 — Sprint 10 후보로 잔존 위험에 기록됨.
